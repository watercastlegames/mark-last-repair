# 새로운 미스터리 테마 만들기

이 문서는 `마크의 마지막 수리` 개발 소스를 바탕으로 새로운 사건을 바이브코딩하는 방법을 설명합니다.

코드를 전부 이해한 뒤 시작할 필요는 없습니다. 먼저 사건의 정답과 공개 순서를 글로 정리하고, 아래 파일별 역할을 AI에게 함께 전달해 한 단계씩 바꾸는 방식을 권장합니다.

## 1. 먼저 사건의 정답표를 만드세요

AI 판정이 흔들리지 않으려면 공개 단서보다 먼저 사건의 확정 사실을 완성해야 합니다.

```text
피해자:
가해자:
사건 장소:
사망 원인:
가해 동기:
숨겨진 진실:

사건 진행:
1.
2.
3.
4.

반드시 참인 사실:
- 

반드시 거짓인 사실:
- 
```

한 문장 안에 복수의 해석이 생기지 않도록 인물의 이름과 역할을 분명히 적으세요. 게임 호스트인 집사를 사건 등장인물로 넣지 않는 것도 중요합니다.

## 2. 네 라운드의 공개 순서를 설계하세요

기본 구조는 라운드마다 다음 내용을 사용합니다.

- 대표 사진 1장
- 사진에서 확인할 단서 3개
- 시간의 방 추가정보 3개
- 집사 질문 최대 3개
- 네 명의 동료 의견

권장 정보 공개 순서:

| 라운드 | 목적 | 예시 |
|---|---|---|
| 1 | 현장과 사망 원인 | 장소, 시신 주변, 가장 이상한 흔적 |
| 2 | 피해자의 행동 | 일정, 소지품, 마지막 동선 |
| 3 | 범행 방법 | 도구, 물질, 준비 흔적 |
| 4 | 범인과 동기 | 과거 기록, 관계, 숨겨진 사건 |

라운드 1에서 범인과 동기를 직접 공개하지 마세요. 반대로 라운드 4까지 정답을 추론할 정보가 부족해서도 안 됩니다.

## 3. 파일별로 사건을 교체하세요

### `assets/js/case-data.js`

다음 데이터를 새로운 사건으로 바꿉니다.

- `CASE_SLUG`: 사건마다 겹치지 않는 영문 ID
- `CASE_OVERVIEW_SUMMARY`: 첫 화면의 짧고 강한 사건 소개
- `CASE_OVERVIEW_DETAILS`: 피해 상황과 초기 의문
- `ROUNDS`: 사진과 단서 3개씩
- `COMPANIONS`: 가상 동료 이름, 역할, 초상화
- `TIME_ROOM_HINTS`: 라운드별 추가정보 3개
- `TIME_ROOM_HINT_TERMS`: AI 동료가 힌트를 반영했는지 검사할 핵심어
- `FALLBACK_HINTS`: AI 연결 실패 때 사용할 안전한 추천 질문
- `ENDING`: 최종 공개할 사건 내막 4단계

### `assets/js/puter-ai.js`

AI가 현재 사건만 판단하도록 아래 부분을 반드시 함께 수정합니다.

- `persona`: 네 동료의 말투와 추리 방식
- `futureTerms`: 아직 공개되지 않은 라운드의 금지 단어
- `visibleTerms`: 라운드별 공개 가능한 단어
- `caseFacts`: 집사 판정에 사용하는 사건의 전체 확정 사실
- `판정 예시`: 새 사건에 맞는 YES, NO, MAYBE, IRRELEVANT 예시
- 최종 동료 의견 프롬프트에 포함되는 사건 역할과 공개 범위

`caseFacts`를 바꾸지 않으면 화면의 사건만 새것이고 AI는 이전 사건의 정답을 기준으로 답하게 됩니다.

### `assets/js/final-evaluation.js`

최종 발의의 여섯 항목을 새 정답으로 교체합니다.

- 피해자
- 가해자
- 장소
- 사망 원인
- 동기
- 숨겨진 진실

각 항목에는 다음 세 가지를 준비하세요.

1. 정확한 정답으로 인정할 표현
2. 부분정답으로 인정할 핵심어
3. 틀렸을 때 보여 줄 정답 기준과 설명

### 사건 ID가 중복되는 다른 파일

새 사건을 별도 랭킹으로 운영하려면 다음 값도 함께 변경합니다.

- `assets/js/state.js`: 로컬 저장 키와 `caseId`
- `assets/js/firebase-ranking.js`: `COLLECTION`, `CASE_ID`

## 4. 이미지를 교체하세요

권장 규격:

- 라운드와 결말 이미지: 16:9, 1536×864 이상
- 동료 초상화: 1:1 정사각형, 768×768 이상
- 모바일에서 중요한 물체는 이미지 가장자리보다 중앙 영역에 배치
- 이미지 안에 작은 글자를 넣지 않기
- 방송 캡처, 유명인의 실명 사진, 권리관계가 불분명한 이미지는 사용하지 않기

기존 파일명을 유지하면 코드 변경을 줄일 수 있습니다. 새 파일명을 사용한다면 `case-data.js`의 경로를 수정하세요.

## 5. AI 공개 범위를 검증하세요

각 라운드에서 동료에게 다음 질문을 해 보세요.

- 아직 공개되지 않은 범인 이름을 먼저 말하지 않는가?
- 시간의 방에서 연 정보가 의견에 반영되는가?
- 이전에 집사에게 받은 답변이 다음 의견에 반영되는가?
- 집사를 범인이나 사건 관계자로 취급하지 않는가?
- 명확한 질문에 무조건 MAYBE로 답하지 않는가?
- 복합 질문과 대명사가 불명확한 질문만 MAYBE로 처리하는가?

오류가 나면 프롬프트를 길게 늘리기보다 `caseFacts`, `futureTerms`, 판정 예시 사이의 모순부터 확인하세요.

## 6. 번들을 다시 만드세요

```bash
npx --yes esbuild assets/js/app-round11.js \
  --bundle \
  --format=iife \
  --target=es2020 \
  --outfile=assets/js/game-round54.bundle.js
```

새 번들 이름을 사용한다면 `index.html`의 `<script>` 주소를 같은 이름으로 변경합니다.

## 7. 로컬에서 확인하세요

```bash
python -m http.server 8080
```

```text
http://localhost:8080/
```

`file://`로 직접 열면 Puter.js가 동작하지 않습니다.

최소 확인 항목:

- 360px 모바일 화면에서 가로 스크롤이 생기지 않는가?
- 모든 팝업에서 뒤로가기와 닫기가 가능한가?
- 단서 3개를 본 뒤 다음 안내가 나타나는가?
- 동료 의견을 다시 열 수 있는가?
- 질문 후 과거 사진과 기록을 다시 볼 수 있는가?
- 새로고침 후 현재 단계가 복구되는가?
- 네 라운드 종료 후 최종 발의와 전체 내막이 나오는가?

## 바이브코딩 요청 예시

아래 내용을 사용하는 AI 도구에 현재 저장소와 함께 전달하세요.

```text
이 저장소는 네 라운드로 진행되는 AI 미스터리 웹게임이다.
CUSTOM_THEME_GUIDE.md를 기준으로 기존 사건을 새로운 사건으로 교체해줘.

새 사건의 정답표:
[피해자, 가해자, 장소, 사망 원인, 동기, 숨겨진 진실 입력]

요구사항:
1. 라운드마다 사진 1장, 공개 단서 3개, 시간의 방 힌트 3개를 만든다.
2. 동료는 현재까지 공개된 정보만 사용한다.
3. case-data.js뿐 아니라 puter-ai.js의 caseFacts와 final-evaluation.js도 함께 수정한다.
4. 집사는 사건 인물이 아니라 게임 호스트다.
5. 명확한 질문에는 YES 또는 NO, 일부만 맞으면 MAYBE, 무관하면 IRRELEVANT로 판정한다.
6. 모바일 360px에서 모든 선택과 결과가 중앙 팝업으로 보이게 유지한다.
7. 수정 후 번들 생성과 문법 검사를 수행한다.
```

## 마지막 권장사항

처음부터 거대한 사건을 만들기보다 네 라운드와 하나의 정답만 완성한 뒤 테스트하세요. 한 사건이 끝까지 안정적으로 진행되면 인물, 이미지, 랭킹과 추가 연출을 확장하는 편이 훨씬 빠릅니다.

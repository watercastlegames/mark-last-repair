DETAILED GAME DESIGN DOCUMENT · v1.0

마크의 마지막 수리

비공식 팬메이드 팀 미스터리 웹게임. 질문으로 정보를 사고, 개인 시간을 걸어 팀의 생존을 만든다.

범위와 권리 원칙: 피의 게임X의 게임 구조에서 영감을 받은 비상업 임시 팬페이지 프로토타입이다. 실제 방송의 문제·사진·영상·음성·대사는 쓰지 않는다. 실제 출연진 이름은 동료 슬롯의 팬메이드 표기로만 사용하고, 초상 대신 독자 제작한 상징 일러스트를 쓴다. 화면에 “비공식 팬메이드 · 방송/제작사와 무관”을 상시 표시한다.

1. 제품 정의와 성공 기준

항목상세
플레이 인원5팀 × 4명까지. MVP에서는 봇 팀 4개 + 사용자 팀 1개로도 테스트 가능.
매치 길이20~30분. 1라운드 30분, 2라운드 20분을 기본값으로 둔다.
핵심 루프사건 이해 → 질문 비용 지불 → 단서 공유 → 팀 합의 → 정답 발의 → 개인 시간으로 생존 판정.
승리 경험정답을 ‘많이 아는 팀’보다, 최소 비용으로 확정하는 팀이 이긴다는 체감.
MVP 성공 기준테스트 팀의 70%가 2라운드까지 도달하고, 질문·시간의 방·발의 중 최소 두 가지 선택을 전략적이었다고 평가.

2. 역할 구성

사용자 플레이어팀 토론, 투표실 질문, 시간의 방 입장, 정답 발의를 선택한다. 모든 행동의 개인 시간은 개별 누적된다.
집사 AI사건 판정기. 설명을 만들거나 대화하지 않는다. 네 가지 고정 답변 하나만 출력한다.
동료 4명토론의 막힘을 푸는 창작 힌트 카드. 정답을 공개하지 않고 질문 형태·관찰 방향만 제안한다.
게임 서버개인 시간, 방 점유, 라운드 상태, 발의 순서, 팀 순위를 권위 있게 계산한다.

동료 팀 — 팬메이드 표기

이름성별 구성조력 역할힌트 규칙
최예선여성 1명정보 정리확정·미확정 사실을 분리한다.
콩진호남성 1명가설 분해큰 추측을 예/아니요 질문으로 쪼갠다.
남출구남성 1명현장 관찰그림의 물건·배치·시간 모순을 재관찰시킨다.
이성민남성 1명시간 전략질문과 시간의 방의 비용 대비 가치를 판단하게 한다.

3. 매치 상태와 전체 흐름

로비 → 1R 사건 공개 → 질문/시간의 방/토론 → 1R 정답 발의 → 1위 확정 → 2R 사건 공개 → 질문/시간의 방/토론 → 순위 확정 → 개인 시간 생존 판정

상태입력 가능 행동종료 조건
사건 공개그림 확대, 팀 토론, 동료 힌트 확인즉시 질문 단계로 이동
추리 진행투표실 입장, 시간의 방 입장, 팀 토론, 정답 발의정답 팀 발생 또는 라운드 종료 시간
1라운드 결과정답 해설의 핵심 키워드 확인1위팀 제외 후 2라운드 시작
2라운드 결과팀 순위와 누적 개인 시간 확인생존 후보/데스매치 대상 산정

1라운드에서 가장 먼저 완전 정답을 발의한 팀은 최종 1위, 2라운드 면제다. 남은 4팀 중 2라운드 첫 완전 정답 팀이 최종 2위다. 질문·시간의 방·오답 발의로 생긴 개인 시간은 라운드가 바뀌어도 초기화하지 않는다.

4. 핵심 시스템 상세

4-1. 사건 공개

팀에게는 사건 개요, 현장 그림 1장, 최종 정답 키워드의 개수만 노출한다. 예: “피해자는 누구인가?”, “왜 죽었는가?”, “숨겨진 진실은 무엇인가?”라는 방향만 안내한다. 키워드 이름·인물 관계·범행 순서는 숨긴다.

4-2. 투표실과 질문 비용

규칙세부
입장개인만 입장. 팀 공용 채팅은 읽을 수 있으나 투표실 화면의 질문/답변은 입장자에게만 우선 노출된다.
질문 수한 번의 입장에 최대 3개. 질문 1개당 개인 시간 +1분.
질문 입력자유 문장. 서버는 질문 원문, 분류 결과, 집사 답변, 비용을 로그로 저장한다.
재입장가능. 다시 최대 3개 질문을 할 수 있으나, 총 질문 수에는 제한을 두지 않는다. 비용이 자연스러운 제한이다.

4-3. 집사 AI — 고정 출력 계약

집사는 답변 뒤에 어떤 이유도 붙이지 않는다. 말풍선·음성·타이핑 연출도 네 문구 밖의 텍스트를 보여 주지 않는다.

분류출력 문자열 (정확히 일치)판정 조건
TRUE네, 그렇습니다.질문의 핵심 전제가 사건의 확정 사실과 일치
FALSE아니요, 그렇지 않습니다.핵심 전제가 확정 사실과 불일치
MIXED그럴 수도 있습니다.복수 전제 중 일부만 맞음, 또는 가능하지만 확정 불가
IRRELEVANT중요하지 않습니다.사건의 최종 키워드 및 인과와 무관

권장 구현: LLM은 질문을 intent + entities + proposition으로 구조화하는 데만 쓰고, 최종 응답은 서버의 판정 테이블이 결정한다. LLM 장애 시에는 의도 분류 사전과 유사 문장 검색으로 폴백한다. 따라서 모델이 정답을 지어내거나 보충 설명을 붙일 수 없다.

4-4. 시간의 방

항목설계
열림 시점각 라운드 시작 30분 후. MVP 테스트는 10분으로 단축 가능.
내용추가 그림 1장과 단서 1~2개. 정답 자체가 아닌 인과를 좁히는 증거.
위험입장 즉시 개인 시간의 흐름이 2배. 한 번에 한 명만 입장. 메모 기능 없음.
협업입장자는 기억한 내용을 사건실에서 구두/텍스트로 전달한다. 캡처는 완벽히 막을 수 없으므로 팬게임의 신사 규칙과 신고 버튼을 병행한다.

4-5. 정답 발의

팀 대표는 구조화 입력으로 발의한다: 피해자 / 가해자 / 장소 / 동기 / 사건 진행 / 사망 원인 / 숨겨진 진실. 서버는 필수 키워드와 인과 규칙을 확인한다. 완전 정답이면 팀 시간을 멈추고, 오답이면 발의자 개인 시간에 +5분을 더한 뒤 계속 진행한다.

5. 동료 힌트 시스템

동료는 집사와 다르다. 집사는 사실만 판정하고, 동료는 플레이어의 현재 행동을 바탕으로 ‘다음 질문의 방향’을 제안한다. 각 동료는 라운드당 1회, 총 4회만 발화한다. 힌트는 자동 재생이 아니라 플레이어가 동료 의견 듣기를 눌러 열람한다.

트리거우선 동료예시 창작 힌트금지
사건 공개 직후최예선“우선 그림에서 확실히 보이는 사실만 세 개 적어 보자.”범인·키워드 언급
‘그럴 수도 있습니다’ 2회콩진호“질문이 너무 넓어. 사망 원인부터 하나씩 나눠 물어보는 게 어때?”집사 답변의 해석 확정
그림 확대를 하지 않음남출구“배전반 말고 바닥과 작업가방 위치도 다시 봐야 할 것 같아.”숨은 단서 위치의 직접 지시
시간의 방 개방이성민“정보는 필요하지만, 지금 들어갈 사람이 2라운드까지 버틸 수 있는지 계산하자.”특정 플레이어에게 희생 강요

실제 인물의 말투 모사는 피하고, 카드 이름과 전략 역할만 구분한다. 힌트 본문은 독자 작성 고정 문장으로 운영한다.

6. 사건 콘텐츠 상세 — 1라운드: 마크의 마지막 수리

초기 공개폭풍우가 치던 밤, 폐극장 배전실에서 전기 수리공 ‘마크 리’가 숨진 채 발견됐다. 바닥은 젖어 있고 차단기는 내려가 있다. 필수 키워드 5개.
운영용 정답 키워드마크 / 그랜트 / 배전실 / 감전 유도 / 12년 전 금화 절도
정답 전말전 경비원 그랜트는 12년 전 금화 절도의 공범이었다. 마크가 발견한 금화 상자 속 기록이 자신을 폭로할 것을 두려워해, 소금물을 이용한 감전 사고로 위장했다.
시간의 방 단서소금물 스프레이가 있는 분장실, 열쇠 복제 비용이 지워진 회계장부, “상자 안 명단을 봤다”는 마크의 음성메시지 파형 이미지.

9. 데이터·서버 설계

Match ├─ teams[]: id, members[], rank, answeredRound ├─ players[]: personalTimeSeconds, questionsAsked, roomVisits, isRepresentative ├─ rounds[]: caseId, status, startedAt, timeRoomOpenedAt ├─ questionLog[]: playerId, rawQuestion, classifiedProposition, response, timeCost ├─ answerAttempts[]: teamId, playerId, structuredAnswer, result, timePenalty └─ roomState: occupiedByPlayerId, enteredAt
  • 서버 권한: 시간 증감, 질문 횟수, 방 입장 잠금, 정답 판정은 반드시 서버에서 처리한다.
  • 실시간: WebSocket 또는 Supabase Realtime으로 팀 채팅·방 점유·순위를 동기화한다.
  • LLM 경계: 클라이언트는 모델 API 키를 갖지 않는다. 서버가 구조화된 질문 분류만 호출하고, 문자열 출력은 자체 테이블에서 선택한다.
  • 감사 로그: 질문 원문과 판정값을 보관해 ‘왜 중요하지 않습니다가 나왔는가’의 운영 검토가 가능해야 한다.

10. 테스트 시나리오 및 완료 조건

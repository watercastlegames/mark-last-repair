9. Puter AI 어댑터와 프롬프트 계약

9.1 동료 의견

SYSTEM: 너는 비공식 팬메이드 추리 게임의 동료 조력자다. 실제 인물의 말투를 모사하지 말고 제공된 게임용 페르소나만 반영한다. CURRENT_PUBLIC_FACTS 밖의 인물, 사건, 숫자, 도구, 정답을 언급하지 마라. 집사에게 입력 가능한 예/아니요 질문 하나를 제안하라. 출력 JSON: {"question":"...습니까?","comment":"...라고 물어보자."} question 45자 이하, comment 80자 이하. USER: ROUND_NO, PERSONA, CURRENT_PUBLIC_FACTS, PREVIOUS_QUESTIONS

응답 후 validateCompanionOutput()에서 JSON 형태, 길이, 질문 부호, 현재 라운드 금지 키워드를 검사한다. 실패하면 라운드별 고정 폴백 문장을 사용한다.

9.2 집사 판정

SYSTEM: 너는 추리 게임 집사다. CASE_FACTS와 CURRENT_REVEAL_SCOPE를 기준으로 사용자 질문을 판정한다. 정확히 다음 JSON만 출력한다. {"answer":"YES|NO|MAYBE|IRRELEVANT"} 이유, 설명, 인사, 추가 필드 금지. 클라이언트 표시 매핑: YES → 네, 그렇습니다. NO → 아니요, 그렇지 않습니다. MAYBE → 그럴 수도 있습니다. IRRELEVANT → 중요하지 않습니다.
공식 기록 주의: Puter 호출은 브라우저에서 수행되므로 응답 자체를 보안 판정 근거로 쓰지 않는다. 서버는 질문 슬롯과 페널티를 먼저 예약하고 최종 정답은 서버 정답 데이터로 별도 판정한다. Puter 응답 변조가 공식 시간·질문 수·최종 정답 판정을 바꿀 수 없어야 한다.

9.3 질문 2단계 저장

  1. question_reserve.asp: 질문 문장을 저장하고 질문 번호·+60초를 확정, questionId 반환.
  2. 브라우저: Puter AI 호출 후 네 허용 카테고리 중 하나로 정규화.
  3. question_commit.asp: questionId와 카테고리 저장. 이미 commit됐으면 기존 결과 반환.
  4. Puter 실패 시 로컬 폴백 판정 후 source="fallback" 저장.

10. Classic ASP JSON API 계약

모든 API는 UTF-8 JSON, Cache-Control: no-store, 성공 시 HTTP 200, 인증 실패 401, 전이 충돌 409, 검증 실패 422, 과다 요청 429, 서버 오류 500을 사용한다.

question_reserve.asp 필수 서버 검증 순서

  1. 로그인 Session 확인.
  2. attempt 소유자와 status='in_progress' 확인.
  3. DB의 current_round와 요청 round 일치 확인.
  4. 단서 3개 확인 여부 확인.
  5. 현재 라운드 완료 질문 수가 3 미만인지 확인.
  6. 질문 1~120자, 빈 문자열·HTML 태그 제거.

17. 보안·개인정보 체크

  • 정답·금지 키워드·최종 해설을 초기 HTML/JS 번들에 넣지 않고 서버 include에서 관리한다.

  • 동료 AI에 현재 라운드 공개 정보만 전달.
  • 운영 오류 응답에 연결 문자열·SQL·토큰·파일 경로를 포함하지 않는다.
